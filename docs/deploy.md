# Deployment Notes

SN POS V2 is a React + TypeScript + Vite app. The production build is generated by:

```bash
npm run build
```

The build script runs `tsc -b && vite build`. The current Vite config uses:

- `@vitejs/plugin-react`
- `@tailwindcss/vite`
- `@` alias mapped to `./src`
- default Vite production output in `dist/`

No custom production build override is required for Vercel staging.

## Required Environment Variables

Set these in Vercel for the staging environment. Do not commit real values.

```bash
VITE_SUPABASE_URL=https://your-project-ref.supabase.co
VITE_SUPABASE_ANON_KEY=your-supabase-anon-key
```

These variables are intentionally prefixed with `VITE_` because they are used by the browser client. Use the Supabase anon key only, never the service role key.

## Vercel Staging Setup

1. Create or open the Vercel project for `sn-pos-v2`.
2. Connect the Git repository and select the staging branch.
3. Use Vercel's Vite defaults:
   - Framework Preset: `Vite`
   - Install Command: `npm install`
   - Build Command: `npm run build`
   - Output Directory: `dist`
4. Add the required environment variables under the staging environment:
   - `VITE_SUPABASE_URL`
   - `VITE_SUPABASE_ANON_KEY`
5. Trigger a staging deployment.
6. Open the staging URL and confirm the login page loads.
7. Log in with an approved test user and run the pre-deploy live QA checklist in `docs/pre-deploy-live-qa.md`.

## Supabase Staging Checklist

- [ ] Auth is enabled for the Supabase project.
- [ ] At least one approved test user exists for staging QA.
- [ ] The database migrations in `supabase/migrations/` are applied in order.
- [ ] Row Level Security is enabled on app tables.
- [ ] Authenticated-user RLS policies exist for:
  - `categories`
  - `products`
  - `product_variants`
  - `customers`
  - `orders`
  - `order_items`
  - `inventory_movements`
  - `store_settings`
- [ ] Storage buckets exist:
  - `products`
  - `store-assets`
- [ ] Storage buckets allow the expected image MIME types:
  - `image/jpeg`
  - `image/png`
  - `image/webp`
- [ ] Storage policies allow authenticated users to read, upload, update, and delete app images in `products` and `store-assets`.
- [ ] Seed or baseline data exists for categories and store settings.
- [ ] If recent migrations changed tables, columns, enums, or relationships, reload the PostgREST schema cache before QA.

## Schema Cache Reload

If staging shows missing columns, stale enum values, or relationship errors after migrations, reload the Supabase schema cache.

From Supabase SQL editor:

```sql
notify pgrst, 'reload schema';
```

Then refresh the staging app and retry the affected flow.

## Pre-Deploy Verification

Run locally before promoting staging:

```bash
npm run lint
npm run build
```

Both commands must pass. Existing lint warnings should be reviewed, but deployment is blocked only by lint or build errors.
